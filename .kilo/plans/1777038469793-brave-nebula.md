# Plan: NetworkX Campus Network Feature Alignment

## Goal
Use the existing NetworkX-based campus network simulation as the source of truth and implement or model the missing campus-network features as far as possible in code.

## Scope
- Preserve the current simulation architecture unless a feature requires a small structural change.
- Model Cisco-like behavior in NetworkX where possible.
- Distinguish between:
  - real configuration evidence in text/config files
  - simulated behavior in Python
  - features that cannot be fully represented in NetworkX and should remain documented as approximations

## Implementation Steps
1. **Inventory current model coverage**
   - Map existing nodes, edges, VLANs, server roles, DHCP logic, ACL logic, and routing behavior.
   - Identify which requested features are already represented in `campus_network_simulation.py`, `config.yaml`, and `notes.txt`.

2. **Add or refine VLAN modeling**
   - Ensure VLANs 101-108, 201-204, and 128 are explicitly represented in the topology model.
   - Add per-switch metadata for access VLAN membership and trunk relationships.
   - If needed, define a clean structure for port-to-VLAN assignment in the graph.

3. **Model trunk links and router-on-a-stick**
   - Represent trunk edges between access/distribution and distribution/core devices.
   - Store allowed VLAN lists and encapsulation details as edge attributes.
   - Add router subinterface metadata for `Gi0/1.101` through `Gi0/1.204` and `Gi0/1.128` if the simulation is meant to reflect the target design.

4. **Improve DHCP modeling**
   - Create one DHCP pool per required VLAN.
   - Enforce correct subnet format and gateway assignment.
   - Update reserved-address logic so the first usable addresses are excluded per the requirement.

5. **Extend inter-VLAN routing representation**
   - Add L3 switch behavior for CORE-SW, DIST-1, and DIST-2 as simulated routing nodes.
   - Model SVIs with VLAN IPs and admin state.
   - Ensure the graph can answer whether an SVI is up and routable.

6. **Formalize ACL and exam-mode behavior**
   - Represent the exam ACL as a named rule set.
   - Preserve the logic for permitting Exam and Email destinations while blocking others.
   - Separate “configured ACL” from “temporarily applied ACL” so the exam-demo requirement can be represented cleanly.

7. **Model server farm services**
   - Attach server IP, mask, gateway, and service flags to the three server nodes.
   - Represent HTTP and HTTPS service status as node attributes.

8. **Add ASA/firewall and STP abstractions**
   - Model ASA inside/outside interfaces, security levels, and ACL behavior as graph metadata.
   - Represent spanning-tree mode, portfast, and system-id settings as switch-level attributes.

9. **Update outputs and reporting**
   - Add a report layer that can emit feature-by-feature evidence from the model.
   - Make it easy to distinguish implemented, partial, and missing features from the current topology state.

10. **Verify against the feature checklist**
   - Re-run file inspection and model inspection to confirm each requested item has evidence.
   - Produce a final feature matrix with implemented, partial, and missing items.

## Verification Criteria
- Each requested feature should have a direct code or config representation, or be explicitly marked as a modeled approximation.
- The final output should still support the current simulation runs.
- The plan should preserve current behavior unless a feature change requires adjustment.

## Risks
- Some items, especially ASA behavior, HTTPS toggles, and Cisco-specific CLI commands, cannot be fully reproduced in NetworkX alone.
- The plan may need to document approximations instead of claiming exact device configuration parity.
